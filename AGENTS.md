# AGENTS.md

Research artifact for the ChatVis paper (LLM-driven ParaView script generation). **Two implementations coexist:** original Jupyter notebooks under `notebooks/` (reference behavior preserved for the paper) and a Python package under `chatvis/` (active reimplementation). Notebook behavior is the source of truth for what the paper claims; the package is the direction of travel and is not byte-identical to it.

## Read first

- The package pipeline is **wired end-to-end** through `chatvis/main.py`. `python -m chatvis.main --help` works; a real run is `python -m chatvis.main --scenario <s> --data-filepath <p> --screenshot-path <p> --username <anl-id>`, which executes prompt-improvement → code-generation → script extraction → `pvpython` → bounded repair loop.
- The old `chatvis/documents/` package was removed. All prompt/data sources now live under `chatvis/v1/`. Do not re-add `chatvis/documents/`.
- `chatvis/v2/` exists on disk but is **untracked** (not in `.gitignore` either — it's a verbatim drop of a different upstream "ChatVis-main" repo containing its own `ChatVis_agent/` and `ChatVis_benchmark/`). Nothing in the active codebase imports from it. Treat it as scratch/reference material; do not import it, do not `git add` it without asking, and do not assume its `AGENTS.md` or `README.md` describe this project.
- Live verification against Argonne's Argo endpoint has not been performed in-session — it requires an ANL username. The orchestrator has been exercised end-to-end with a stubbed LLM and a real `pvpython` subprocess. The LLM wire format has not been confirmed against Argo.
- `data/benchmark/` is **untracked** and holds per-scenario reference outputs (`<scenario>.py`, `<scenario>-screenshot.png`, `prompt.txt`, sometimes the dataset itself). Treat it as ground truth for "what a successful run looks like" but do not commit it.
- `out/` is generated by the package CLI and `scripts/run-all-scenarios.sh`; safe to delete.

## Running the package pipeline

Single scenario (Linux, `pvpython` on `PATH`):

```bash
uv run python -m chatvis.main \
    --scenario stream-glyph \
    --data-filepath data/disk.ex2 \
    --screenshot-path /tmp/stream-glyph.png \
    --username <your-anl-id>
```

All five scenarios in batch with summary table:

```bash
scripts/run-all-scenarios.sh <your-anl-id> [extra chatvis flags ...]
```

Writes `out/<scenario>.{png,py,log}`; continues past individual failures; exits non-zero iff any scenario failed. Extra args after the username pass through to every `chatvis` invocation (e.g. `--log-level debug`, `--max-repair-attempts 10`).

**Path derivation**: the orchestrator derives the generated script path from `--screenshot-path` by replacing the suffix with `.py`. Script and screenshot land side-by-side.

**Manual verification of a successful run** (there is no automated screenshot diff):

1. CLI exits `0`.
2. Visually compare the generated screenshot with `data/benchmark/<scenario>/<scenario>-screenshot.png` (or with the corresponding notebook).
3. Inspect the generated `.py` against `data/benchmark/<scenario>/<scenario>.py` for major divergence.

**ParaView version drift risk**: the notebooks were authored against ParaView 5.12; `environment.yaml` pins 5.13.3; the developer machine may have something else (e.g. 6.1.1 from miniconda). Snippets in `chatvis/v1/documents/code.py` may need updates if the local pvpython rejects them. Flag failing scenarios in a follow-up rather than patching snippets ad-hoc inside an orchestrator session.

## Layout

- `chatvis/` — installable package (Python 3.14).
    - `cli.py` — argparse. Flags: `--scenario`, `--data-filepath`, `--screenshot-path`, `--model` (only `gpt4o` today), `--username` (Argo auth), `--endpoint`, `--pvpython` (default `shutil.which("pvpython")`), `--max-repair-attempts` (default `5`), `--log-file`, `--log-level`.
    - `main.py` — orchestrator. `run_pipeline()` runs the four stages and the bounded repair loop; `check_data()` (`chatvis/main.py:70`) validates `--data-filepath` exists and warns (does not error) on scenario↔file mismatch via `_EXPECTED_DATA_BY_SCENARIO`. Exit codes: `0` clean run, `1` repair-loop exhausted or no python block returned, `2` pre-flight configuration error.
    - `llm.py` — `OpenAIModel` (reproducible defaults: `seed=42`, `temperature=0.0`, `top_p=1.0`, `n=1`); the three pipeline-stage helpers `improve_prompt`, `generate_code`, `improve_code` (the last one takes both `errors=` and `stdout=` so the LLM can see anything the previous script `print()`ed, e.g. PointData array names — always thread stdout through, do not drop it); `parse_response`, `_substitute_paths`, `_FEW_SHOT_KEY_BY_SCENARIO` (binary toggle between the two example families so the few-shot is never self-referential).
    - `utils.py` — `extract_python_code(text) -> list[str]` returns every fenced `python` block; `extract_error_messages(stderr) -> list[str]` first looks for Python tracebacks anchored on `Traceback (most recent call last):` (so generic VTK ANSI-coloured warnings don't trigger the repair loop) and, only if no traceback is present, falls back to a narrow allowlist of VTK/ParaView silent-failure substrings (`Contour array is null`, `Error reading binary data`, `Unable to update`, `Update failed`, `no input`, `Cannot show the data`) so that pvpython runs which exit 0 with empty geometry still re-enter the repair loop. **This is a clean reimplementation; do not assume parity with the notebook helpers of the same name (see "Notebook gotchas").**
    - `logger.py` — `configure_logging(log_to_file, level)` installs handlers on the `chatvis` root logger with UTC timestamps; returns the log file path or `None`.
- `chatvis/v1/` — **canonical active target** for the refactor.
    - `v1/script.py` — `derive_script_path(screenshot_path)`, `first_python_block(llm_response)` (raises `ValueError` if no fenced block; debug-logs when discarding extras), `write_script(code, path)`.
    - `v1/pvpython.py` — `run_pvpython(pvpython_path, script_path)`. Pre-flight validates the binary exists and is executable and that the script exists, before spawning. `check=False`; the orchestrator treats an extracted traceback as the failure signal, not the return code (pvpython can exit non-zero with no Python error, and vice versa).
    - `v1/documents/code.py` — `CodeSnippet(StrEnum)`: ParaView code examples used as LLM context. Snippets contain `<input_path>` / `<output_path>` angle-bracket sentinels intended for **LLM** substitution, not Python `str.format`.
    - `v1/documents/prompts.py` — `GeneratedPrompt(StrEnum)`: **few-shot example pairs only**. Module-level `_COMMON_INPUT` / `_COMMON_OUTPUT` constants are intentionally shared by `ML_DVR`, `ML_ISO`, `ML_SLICE_ISO`, `STREAM_GLYPH`; only `POINTS_SURF_CLIP_*` is distinct. Sentinels are angle-bracket (`<input_path>`, `<output_path>`) and substituted by `chatvis.llm._substitute_paths`.
    - `v1/prompts/user_prompts.py` — `UserPrompts(StrEnum)`: **the user-facing scenario description for each scenario**. Uses Python `str.format` placeholders `{input_path}` / `{output_path}` (different convention from `GeneratedPrompt` — see below).
    - `v1/prompts/{code_generation,code_improvement,prompt_improvement}.py` — `StrEnum`s with `SYSTEM_PROMPT` and `USER_PROMPT` members. Placeholders are Python `str.format`: `{user_input}`, `{example_user_input}`, `{example_user_output}` for prompt improvement; `{prompt}` for code generation; `{errors}`, `{script}`, `{prompt}` for code improvement.
- `notebooks/` — five scenario notebooks. ~90% near-duplicate scaffolds; duplication is **intentional for paper reproducibility** — do not refactor into a shared module without asking.
- `data/` — three datasets at the root (`ml-100.vtk`, `disk.ex2`, `can_points.ex2`) and the untracked `data/benchmark/` reference set.
- `scripts/run-all-scenarios.sh` — batch driver, see above.
- `environment.yaml` — **conda env whose only job is to put `pvpython` 5.13.3 on $PATH** when the host has no system ParaView. Single explicit dep (`paraview=5.13.3`, conda-forge); everything else is a transitive pin resolved by conda. Completely separate from the uv toolchain: the package itself still runs under uv-managed Python 3.14, this env just supplies the `pvpython` binary the orchestrator shells out to. Not consumed by `uv sync`.
- `dist/`, `chatvis.egg-info/`, `.ruff_cache/`, `out/` — build/tool/run artifacts; do not commit.

## Prompt source-of-truth split (subtle, easy to break)

Two conventions deliberately coexist in `improve_prompt`:

| Source                                                     | Substitution                                                           | Used for                                                       |
| ---------------------------------------------------------- | ---------------------------------------------------------------------- | -------------------------------------------------------------- |
| `UserPrompts[<SCENARIO>]`                                  | `.format(input_path=..., output_path=...)`                             | the user's own scenario description (the thing being improved) |
| `GeneratedPrompt[<KEY>_INPUT]` and `_OUTPUT`               | `_substitute_paths(...)` on `<input_path>` / `<output_path>` sentinels | the few-shot example pair                                      |
| `CodeSnippet` values inside `CodeGeneration.SYSTEM_PROMPT` | LLM substitutes the sentinels itself                                   | ParaView snippets shown to the LLM                             |

Do not "unify" these without checking every call site. In particular, applying `.format` to a string carrying `<input_path>` sentinels is a silent no-op; applying `_substitute_paths` to a string carrying `{input_path}` placeholders leaves the placeholder intact and the downstream `.format` will then either explode or substitute the wrong thing.

`_FEW_SHOT_KEY_BY_SCENARIO` enforces non-self-referential pairing: scenarios in the stream-glyph family use `POINTS_SURF_CLIP` as their few-shot; `points-surf-clip` uses `STREAM_GLYPH`. Editing this dict to a self-referential mapping silently shows the LLM its own answer.

## Toolchain (Python 3.14, uv-managed)

- `pyproject.toml` requires `python>=3.14`; `.python-version` pins `3.14`. `.venv/` at repo root is the working env. Dependencies: `openai>=2.41` (only). Dev: `jupyter>=1.1.1`, `pytest>=9.0.3`. `pydantic` is still present transitively via `openai`; nothing in `chatvis/v1/` imports it.
- **Pre-commit pins Python 3.13** (`.pre-commit-config.yaml:2`: `default_language_version: python: python3.13`) even though the project targets 3.14. Hooks run under 3.13. Do not "fix" without confirming.
- Pre-commit hook order: `pre-commit-hooks` → `isort` → `ruff-format` → `ruff-check` → `bandit` → **local `prettier`** (uses system-installed `prettier`; will fail if not on PATH). Run via `pre-commit run --all-files`. No `make lint` target.
- `[tool.isort]` is set to `profile = "black"`, `line_length = 88` so isort doesn't fight ruff-format's wrapping. No `[tool.ruff]` config — ruff runs on defaults. **If you re-introduce an isort↔ruff-format loop on imports, check `pyproject.toml` first; it's the most common cause.**
- `tests/` exists and is run with `uv run pytest tests -q`. Coverage is intentionally limited to pure helpers in `chatvis.utils`, `chatvis.v1.script`, and the pure helpers of `chatvis.llm` (`_substitute_paths`, `parse_response`, `_FEW_SHOT_KEY_BY_SCENARIO` invariants). `OpenAIModel`, the three pipeline-stage helpers, `run_pvpython`, and `run_pipeline` are NOT covered — they need either a live LLM/`pvpython` or a non-trivial stub. Bandit excludes `tests/` (B101 `assert_used` is unavoidable in pytest).
- No CI workflows.
- Existing ruff failures in `notebooks/*.ipynb` (~17 of them) are **pre-existing and intentional** — those cells are paper artifacts, not active code. Do not auto-fix them.

## Notebook gotchas (all five still present)

- **Hard-coded macOS author paths** appear in three places per notebook: the `extract_python_code` output filename, the `user_input` prompt text, and the `SaveScreenshot` path inside `code_to_save`. All point at `/Users/tanwimallick/Documents/Paraview/generated_code/`. Edit all three per notebook to run locally.
- **macOS-only `pvpython` path prepend**: `"/Applications/ParaView-5.12.0.app/Contents/bin:$PATH"`. No-op on Linux; ensure `pvpython` is otherwise on `PATH`.
- **`OpenAI(api_key="")`** in notebooks — no env-var loading. Paste key directly, do not commit. The package code path uses Argo via `--username` and does not touch this.
- **Dead `import ollama`** in `ml-dvr`, `ml-iso`, `ml-slice-iso`. Notebook will `ImportError` without `pip install ollama` (not in `pyproject.toml`).
- **`extract_python_code` notebook bug**: `return filename` sits inside the `for` loop, so only the first python block is ever saved. Present in all five. **The package version in `chatvis/utils.py` is a clean reimplementation that returns `list[str]`; do not assume parity.**

## Per-notebook behavioral differences (verify before editing)

- `stream-glyph.ipynb` — active `while errors:` correction loop. `gpt-4` for both generation and repair.
- `points-surf-clip.ipynb` — active `while errors:` correction loop. `gpt-4-turbo` for generation, `gpt-4` inside the loop.
- `ml-dvr.ipynb` — `while errors:` block is **commented out**; runs exactly one generation attempt. `gpt-4o` for prompt generation, `gpt-4-turbo` for code generation.
- `ml-iso.ipynb` — no correction loop, but the `subprocess.run` / `extract_error_messages` block is **literally duplicated** across two adjacent cells. Likely a copy-paste artifact; preserve as-is unless asked. `gpt-4o` then `gpt-4-turbo`.
- `ml-slice-iso.ipynb` — no correction loop. Models are swapped vs the others: `gpt-4-turbo` for prompt generation, `gpt-4o` for code generation.

The package collapses all of the above into a single `--model gpt4o` against Argo with a bounded repair loop. Do not chase the per-notebook model assignments into the package without asking.

## Conventions

- The package targets **Argonne's Argo API** — an OpenAI-compatible endpoint at `https://apps.inside.anl.gov/argoapi/v1` (`chatvis/cli.py:DEFAULT_ENDPOINT`). Authentication is by ANL username passed as `api_key` to the OpenAI `Client`. **Not** plain OpenAI. Do not add `OPENAI_API_KEY` loading.
- `--model` choices are restricted to `["gpt4o"]` (`chatvis/cli.py:MODELS`). Adding a new model means updating the CLI choices and confirming Argo accepts it.
- Logging: every submodule uses `logging.getLogger(__name__)`. `chatvis/logger.py` configures the `chatvis` root logger and disables propagation.
- When porting more notebook behavior, follow the migration target: prompts in `chatvis/v1/prompts/`, data in `chatvis/v1/documents/`, orchestration in `chatvis/llm.py` and `chatvis/main.py`.

## Build / setup

- `Makefile`:
    - `make create-dev` → `pre-commit install && pre-commit autoupdate && rm -rf env && uv sync`.
    - `make build` → derives version from the highest git tag, then `uv build`, then `uv pip install dist/*.tar.gz`. Tag `0.1.0` exists, so this currently works; if you ever delete all tags it will pass an empty version to `uv version` and fail.
- Notebooks are committed **with output cells** (large base64 PNGs). Re-running rewrites execution counts and outputs — inspect `git diff` carefully and ask before committing notebook output churn.
- Avoid `jq` / raw-JSON edits to `.ipynb` files. Use Jupyter or `nbconvert`.
- `README.md` covers both the package flow (primary) and the notebook flow (paper reproducibility). It is intentionally lighter than this document — if it conflicts with `AGENTS.md`, treat `AGENTS.md` as authoritative.

Blockers in chatvis/sources.py
================================

File reviewed: chatvis/sources.py (103 lines, three module-level dicts:
CODE_SOURCES, INPUT_SOURCES, PROMPT_SOURCES).

The dicts are intended as reusable building blocks (code primers, user-input
prompts, and system prompts) for driving an LLM to generate ParaView scripts.
The issues below would all prevent or mislead another user trying to consume
them.


HIGH SEVERITY — would cause runtime failure or wrong behavior
-------------------------------------------------------------

1. Hard-coded absolute macOS paths baked into prompts and code (lines 70,
   77, 83, 95, 101).
     - "/Users/tanwimallick/Documents/Paraview/generated_code/disk.ex2"
     - "/Users/tanwimallick/Documents/Paraview/generated_code/stream-glyph-screenshot.png"
     - "/Users/tanwimallick/Documents/Paraview/generated_code/points-surf-clip-screenshot.png"
   The /Users/tanwimallick/... path does not exist on any other machine, so:
     * Any script generated from these prompts will fail to read the input
       data and fail to write the screenshot.
     * code_to_save (line 70) writes the file unconditionally — on a fresh
       checkout pvpython will raise because the parent directory is missing.
   The repo's own AGENTS.md flags this as the CRITICAL issue for the
   notebooks; the same problem has been carried into sources.py verbatim
   instead of being parameterized.

2. CODE_SOURCES["code_to_read"] (line 6) contains a literal `<path>` token:
       ml100vtk = LegacyVTKReader(FileNames=<path>)
   This is not valid Python and not a documented template placeholder. A
   consumer has no signal whether to:
     * format-string substitute it,
     * regex replace it, or
     * treat it as illustrative pseudocode.
   No helper, no docstring, no example call site explains the contract.

3. CODE_SOURCES["code_to_color_transfer_function"] and
   "code_to_opacity_transfer_function" (lines 36, 41) use the bare names
   `min` and `max` as if they were numeric scalars. As emitted, this code
   shadows Python builtins and ParaView will receive `<built-in function
   min>` instead of a float — TypeError at runtime. Same placeholder
   problem as #2: no convention for how to substitute real values.

4. CODE_SOURCES["code_to_clip"] (line 28) references `delaunay3D` as the
   Input, but no snippet in CODE_SOURCES defines `delaunay3D`. Any script
   assembled from these primers alone will hit a NameError. The dependency
   on an upstream Delaunay3D filter is implicit and undocumented.

5. INPUT_SOURCES["ml_dvr"] and PROMPT_SOURCES["ml_dvr"] (lines 75, 89) are
   mislabeled. Both describe the stream-glyph scenario (disk.ex2,
   streamlines of V, cone glyphs, stream-glyph-screenshot.png). The
   repo has a distinct ml-dvr scenario (data/benchmark/ml-dvr/) whose
   ground truth is direct volume rendering of ml-100.vtk — completely
   different inputs and operations. Anyone keying off "ml_dvr" will get
   the wrong prompt.


MEDIUM SEVERITY — would mislead or block reuse
----------------------------------------------

6. Coverage gap: only one scenario ("ml_dvr", which is actually
   stream-glyph) is present in INPUT_SOURCES/PROMPT_SOURCES. The repo
   defines five canonical scenarios (ml-dvr, ml-iso, ml-slice-iso,
   points-surf-clip, stream-glyph — see data/benchmark/). A user
   following the README expecting parity with the five notebooks will
   find four scenarios silently missing.

7. Dict-key naming inconsistency between this file and the rest of the
   repo. Scenario directories use hyphenated kebab-case
   ("ml-dvr", "stream-glyph"), but the dict keys use snake_case
   ("ml_dvr"). With no documented mapping, a caller cannot reliably
   look up "the prompt for scenario X" without string munging.

8. f-string prefixes on lines 75 and 89 (`f"""..."""`) are pointless —
   neither string contains `{}` interpolation expressions. This signals
   to readers that substitution is happening when it is not, which
   compounds the placeholder confusion from #1–#3.

9. The 'ml100vtk' identifier in code_to_slice (line 11) and
   code_to_contour (line 21) hard-codes a variable name that only makes
   sense if code_to_read was used and the file was an ml-100 dataset.
   Combining code_to_slice with, e.g., the disk.ex2 prompt will produce
   a NameError. The implicit ordering/coupling between snippets is
   nowhere stated.

10. No module docstring, no per-key docstrings, no type beyond
    `dict[str, str]`. A consumer cannot tell:
      * which CODE_SOURCES keys are meant to be concatenated vs.
        used standalone,
      * which placeholders (`<path>`, `min`, `max`, `delaunay3D`,
        `ml100vtk`, `renderView`) must be substituted before use,
      * which scenarios INPUT_SOURCES/PROMPT_SOURCES are expected
        to grow to cover.


LOW SEVERITY — quality / maintainability
----------------------------------------

11. Both snippets and prompts contain trailing/leading whitespace from
    triple-quoted strings (every value starts with "\n"). Most
    downstream uses will want stripped text; the file forces every
    caller to `.strip()` or `dedent`.

12. The `from paraview.simple import *` line is repeated inside
    several CODE_SOURCES values (lines 3, 9, 18). If a consumer
    concatenates snippets to build a full script, they will end up
    with duplicate star-imports.

13. There is no test, no example, and no `__all__`. The file is not
    imported anywhere in the repo (grep across the workspace finds
    zero importers), so the intended public API is undefined.


Suggested minimum fixes to unblock external use
-----------------------------------------------

a. Replace all hard-coded /Users/tanwimallick/... paths with explicit
   placeholders (e.g. `{input_path}`, `{screenshot_path}`) and document
   the substitution contract, OR convert the values to functions that
   take paths as parameters.

b. Replace `<path>`, bare `min`/`max`, `delaunay3D`, and `ml100vtk`
   with the same explicit placeholder scheme, and add a short module
   docstring listing every placeholder a caller must fill.

c. Rename INPUT_SOURCES["ml_dvr"]/PROMPT_SOURCES["ml_dvr"] to
   "stream_glyph" (or "stream-glyph") so the key matches what the
   string actually describes; add the four genuinely missing
   scenarios.

d. Drop the redundant f-string prefixes on lines 75 and 89.

e. Add a docstring or README section showing one end-to-end example
   of assembling CODE_SOURCES snippets into a runnable script, so the
   ordering and variable-name coupling between snippets is explicit.
